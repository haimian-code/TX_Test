# 笔试题：肉鸽小游戏 · 流派强弱模拟验证器

> 技术栈要求：**Godot 4.x（GDScript 或 C#）**  
> 提交形式：Git 仓库（含完整工程 + README + 设计文档）

---

## 一、题目背景

我们正在做一款 AI 辅助生成的 **肉鸽（Roguelike）小游戏**。AI 会持续产出新的策划案：新角色、新技能、新道具、新关卡、新怪物。  
策划案的"好不好玩"很大程度取决于 **数值平衡** 和 **流派可行性**——但人工逐个验证成本极高。

因此我们需要一个 **轻量的端侧模拟验证工具**：
- 能快速加载一份策划案配置（关卡 / 怪物 / 角色 / 技能 / 道具）；
- 能跑一场战斗模拟，输出战斗结果（胜负、耗时、伤害曲线、剩余血量等）；
- 能通过 **外部策略 API**，让"AI 决策器"或"测试脚本"动态控制角色的加点、技能释放、道具使用，从而批量验证不同流派的强弱。

> 一句话：**做一个 Godot 写的"流派模拟器"，让策划/AI 可以脚本化地跑战斗、看数据。**

---

## 二、题目要求

### 2.1 你需要自己设计的东西（开放部分）

我们 **不会** 给你策划案、不会给你数据格式、不会给你 API 规范。这部分是考察的重点：

#### ① 一份"小而自洽"的策划案（设计文档 `design.md`）
至少包含：
- **世界观/玩法循环**：1 段话讲清楚"玩家在干什么"。
- **角色系统**：至少 1 个可玩角色，包含基础属性（如 HP/攻击/防御/暴击/速度等，自定义）、**属性加点机制**（升级/技能点……自定义）。
- **技能系统**：至少 3 个技能，包含冷却、消耗、效果（伤害/治疗/Buff/位移……自定义）；要有"技能释放时机会影响强弱"的设计。
- **道具系统**：至少 3 件道具，要有"组合产生质变"的潜力（体现肉鸽的 build 感）。
- **怪物系统**：至少 2 种怪物（如普通怪、精英怪），有差异化属性/行为。
- **关卡结构**：至少 1 个关卡，定义怪物出场顺序/数量。
- **至少 2 套"流派"假设**：例如「暴击流」vs「持续输出流」vs「召唤流」，说明你期望哪种更强、为什么，**这将作为模拟器要去验证的命题**。

#### ② 一份"数据格式规范"（`schema.md` 或写在 design.md 里）
你需要定义：
- 关卡 / 怪物 / 角色 / 技能 / 道具的 **配置文件格式**（推荐 JSON / TOML / Godot Resource，自选）；
- 至少提供 **1 份完整可加载的示例配置**，放在 `data/` 目录下。

#### ③ 一份"流派策略 API 规范"（`api.md` 或写在 design.md 里）
你需要定义模拟器如何被外部策略控制，包括但不限于：
- **初始加点策略**：开局如何分配属性点 / 选择天赋。
- **战斗中决策**：每个决策时机（回合开始 / 每个 tick / 技能 CD 好了），策略需要返回什么动作（释放哪个技能、用哪个道具、是否走位等）。
- **接口形式**：可以是 GDScript 接口类、可以是 JSON-RPC、可以是 HTTP、可以是读外部脚本文件……**自选并说明理由**。
- 至少给出 **2 个不同流派的策略实现样例**，对应你在策划案里提的"流派假设"。

> ⚠ 评分提示：我们更看重 **数据格式与 API 的扩展性、清晰度、可读性**，而不是"功能多"。简单但自洽 > 复杂但混乱。

### 2.2 你需要实现的端侧功能（必做部分）

使用 **Godot** 实现一个可运行的 Web/桌面端模拟器：

#### ① 配置加载
- 启动时（或通过 UI）加载你定义的关卡 / 怪物 / 角色 / 技能 / 道具配置。
- 加载失败要有清晰的报错提示，不允许直接崩溃。

#### ② 场景渲染（可以非常简陋 甚至命令行文字输出也可）
- 在场景中显示：玩家角色 1 个、怪物若干（按关卡定义出场）。
- 美术资源 **随便用**：色块、Godot 内置图标、免费素材都行，**不考察美术**。
- 需要能直观看到：HP 条、伤害飘字（或 log 也可）、技能释放提示。

#### ③ 战斗模拟逻辑
怪物按关卡定义 **排队** 依次上场和玩家碰撞战斗。需要同时支持以下两种模式（可通过 UI 或配置切换）：

- **回合制模式**：双方按速度/先攻顺序轮流行动，每回合一方做一次决策。
- **实时模式（Tick 制）**：以固定时钟周期（如 0.1s / tick）推进，技能 CD、Buff、攻击间隔等都按 tick 结算。

> 战斗逻辑可以**简化**：怪物 AI 可以是"无脑普攻"，不要求复杂寻路、走位、技能链。重点是**框架要能跑通**。

#### ④ 流派策略 API 接入（核心考察点）
- 玩家角色的所有决策（加点、技能、道具）都必须 **通过你定义的策略 API** 来获取，**不允许在战斗逻辑里硬编码**。
- 必须能 **替换策略实现**，跑同一个关卡得到不同结果。
- 提供至少 2 套样例策略对照运行。

#### ⑤ 战斗结果输出
战斗结束后输出至少：
- 胜负、总耗时（回合数 / tick 数 / 秒数）；
- 玩家剩余 HP、各技能释放次数、总伤害 / 承受伤害；
- 伤害随时间的简单曲线（图表或纯文本表格都可）；
- 支持导出为 JSON / CSV，便于批量分析。

#### ⑥ 批量模拟（必做的最低形态）
- 提供一个入口（命令行参数 / 调试按钮 / 简单 UI 都行），可以连续跑 N 次同一关卡（每次随机种子不同），统计胜率、平均耗时。
- 这是"流派强弱验证"的核心场景，必须能跑。

### 2.3 加分项（自由发挥）

按完成度加分：

- 🌟 **Web 部署**：导出为 HTML5 并部署到任意可访问的页面，README 给出链接。
- 🌟 **可视化对比面板**：UI 上能并排展示两套策略的胜率/伤害曲线对比图。
- 🌟 **外部进程策略**：通过 stdio / HTTP / WebSocket 让 Python 等外部程序当策略大脑（方便接 LLM）。
- 🌟 **可重放（Replay）**：保存一场战斗的完整事件流，能原样回放。
- 🌟 **随机词条/装备词缀** 的肉鸽感设计，并能在批量模拟中体现。

## 四、常见问题（FAQ）

**Q1：必须用 Godot 吗？我更熟 Unity / Cocos / 纯前端 Canvas。**  
A：均可，按需自选。

**Q2：策划案我没灵感怎么办？**  
A：可以参考《杀戮尖塔》《吸血鬼幸存者》《以撒》《雨中冒险》等做一个**极简子集**。我们不要求"好玩"，只要求"自洽 + 能验证流派强弱"。

**Q3：战斗逻辑太简单会不会扣分？**  
A：不会。简单清晰 > 花哨混乱。我们**优先看**：策略 API 是否真正能驱动不同流派、批量模拟是否跑得通、数据格式是否扩展性好。

**Q4：可以用 AI 辅助写代码吗？**  
A：可以，**鼓励使用**。但请确保你**理解每一行代码**——我们会在面试环节就你的代码细节、设计取舍进行追问。

**Q5：时间不够全部做完怎么办？**  
A：**优先级**：策划案 + 数据/API 规范 → 单次战斗能跑通（任选一种模式） → 策略 API 解耦 → 双模式 → 批量模拟 → 加分项。  
按这个顺序做，做到哪儿算哪儿，**写清楚 README 说明已完成范围** 比"假装做完"更加分。

**Q6：策略 API 一定要支持外部进程吗？**  
A：不需要。最低要求是"GDScript 内换一个类/脚本就能切换策略"。外部进程是加分项。

## 五、我们最希望从这道题看到什么

1. **你能把一个开放命题拆解成可执行的工程**（设计 → 数据 → 接口 → 实现 → 验证 闭环）。
2. **你写的代码我们读起来很舒服**：分层清晰、命名得体、关键路径有注释、错误处理不偷懒。
3. **你的 API 设计能"经得起未来的需求"**：等下次 AI 又生成一个新职业、新机制，你的工具能不能不改核心、只加配置/策略就跑起来？这就是我们想要的扩展性。
4. **你用工具回答了一个设计问题**：README 末尾的"设计自评"——你预期 A 流派强，跑完发现是 B 强，然后你怎么解释？这是工程师 × 策划的复合能力体现。
